## Workflow по @tasklist.md для разработки по @vision.md

- **Цель**: разрабатывать функциональность строго по плану из @tasklist.md, реализуя видение из @vision.md.
- **Принцип**: KISS. Минимальный инкремент, понятные решения, без лишней сложности.

### Итеративный цикл (каждая задача @tasklist.md)
1) **Согласование решения**
   - Предложить краткий план и примеры кода (фрагменты/интерфейсы/сигнатуры).
   - Формат предложения:
     - Цель итерации
     - Изменения по файлам
     - Примеры кода
     - Критерии приемки
   - Дождаться подтверждения.
2) **Реализация**
   - Реализовать согласованный минимум. Без самовольных отступлений от @tasklist.md.
3) **Проверка**
   - Локальные проверки/тесты (если есть), краткий отчёт и дифф ключевых изменений.
4) **Подтверждение**
   - Запросить подтверждение результатов. Остановиться до ответа.
5) **Обновление прогресса**
   - Обновить @tasklist.md: статус, заметки по прогрессу.
   - Отметить выполненные подпункты как завершённые.
6) **Коммит**
   - Сделать коммит в репозиторий.
   - Сообщение: "feat/fix/docs: кратко что сделано (task: <id/название>)".
7) **Переход**
   - Запросить согласование перехода к следующей итерации/задаче.

### Запреты и оговорки
- Не изменять план без согласования. Любые отклонения — через шаг согласования.
- Если есть блокер/неопределённость — задать 1-2 уточняющих вопроса и остановиться.
- Держать изменения атомарными и обратимыми.

### Коммуникация короткими статусами
- Перед началом: что предполагается сделать.
- После реализации: что сделано и где смотреть код (файлы/фрагменты).
- После проверки: результаты и готовность к подтверждению.
